# Аналитика продукта — Генератор жалоб Химки

## Обзор

Сбор данных через **PostHog SDK** (браузерный, client-side).
Данные собираются с **22 марта 2026**.
Проект: `My App` в PostHog Dashboard.

### Конфигурация PostHog

| Параметр | Значение | Комментарий |
|---|---|---|
| `person_profiles` | `never` | Анонимный режим — нет persistent user ID |
| `autocapture` | `false` | Только ручные события |
| `capture_pageview` | `true` | `$pageview` пишется автоматически |
| API host | `us.i.posthog.com` | |

**Важно:** из-за `person_profiles: never` каждая сессия — отдельный анонимный пользователь. Retention, cohort-анализ и cross-session трекинг недоступны. Это сознательное решение — продукт не требует авторизации и ориентирован на приватность.

---

## Основная воронка

```
$pageview → problem_selected → data_entered → step2_reached → complaint_sent
```

| Шаг | Событие | Ожидаемое падение |
|---|---|---|
| Открыл сайт | `$pageview` | — |
| Выбрал первую проблему | `problem_selected` | Небольшое (сайт целевой) |
| Прошёл валидацию данных | `data_entered` | Возможно часть уходит на модалку |
| Перешёл на шаг 2 | `step2_reached` | — |
| Нажал отправить | `complaint_sent` | Основная метрика конверсии |

**Реальные данные (25 марта 2026, первые 24ч):**
- Уникальных пользователей: 29
- Дошли до step2_reached: 27 (93%)
- Отправили жалобу: 19 (65.5% от всех, 70% от дошедших до шага 2)

## Все события

### `$pageview`
Автоматически PostHog при загрузке страницы.
**Использование:** знаменатель воронки, уникальные пользователи.

---

### `problem_selected`
Срабатывает **один раз за сессию** — когда пользователь выбирает **первый** чип проблемы (переход `sel.size: 0→1`). Повторные клики по другим проблемам не пишутся.

| Свойство | Тип | Описание |
|---|---|---|
| `first_problem` | string | ID первой выбранной проблемы |

**Использование:** показывает сколько людей начали взаимодействие. Разрыв между `$pageview` и `problem_selected` = bounce без вовлечения.

---

### `data_entered`
Срабатывает когда пользователь проходит валидацию личных данных и переходит к шагу 2.

| Свойство | Тип | Описание |
|---|---|---|
| `skipped` | boolean | `true` — нажал «Без ФИО», `false` — ввёл данные |

**Использование:** доля пользователей, пропускающих персонализацию — влияет на качество текста жалобы.

---

### `step2_reached`
Срабатывает при нажатии «Далее» и успешном переходе на экран отправки (шаг 2).

| Свойство | Тип | Описание |
|---|---|---|
| `problems` | string[] | Список ID выбранных проблем |
| `problems_count` | int | Количество проблем |
| `tone` | string | `formal` / `informal` |
| `length` | string | `short` / `medium` / `long` |

**Использование:** основной чекпоинт флоу. Разбивка по `problems` показывает самые популярные проблемы жильцов.

---

### `ai_generated`
Срабатывает когда OpenRouter успешно вернул текст жалобы.
Может срабатывать **несколько раз за сессию** — при каждой перегенерации (смена тона, длины, кнопка «Ещё вариант»).

| Свойство | Тип | Описание |
|---|---|---|
| `problems` | string[] | Список ID выбранных проблем |
| `problems_count` | int | Количество проблем |
| `tone` | string | `formal` / `informal` |
| `length` | string | `short` / `medium` / `long` |
| `model` | string | Название модели OpenRouter |

**Важно:** событие стреляет на **шаге 1** (до `step2_reached`), поэтому его нельзя ставить обязательным шагом воронки. Использовать как отдельный инсайт: `ai_generated / $pageview` = % сессий где ИИ сработал.

---

### `ai_failed`
Срабатывает когда ИИ-генерация не удалась и пользователь получил шаблонный текст.

| Свойство | Тип | Описание |
|---|---|---|
| `problems_count` | int | Количество выбранных проблем |
| `reason` | string | `api_error` — сервер вернул не-200; `exception` — сетевая ошибка / таймаут |

**Что видит пользователь при `ai_failed`:** анимированный скелетон "ИИ пишет жалобу…" → через 1-3 сек появляется шаблонный текст. Визуально незаметно — сообщения об ошибке нет.

**Что такое шаблонный текст:** генерируется локально функцией `gen()` из `texts.js`. Уникален — использует seeded random для выбора из множества вариантов фраз, подставляет имя/квартиру/дату/адресата. Качество ниже ИИ-текста (не связывает проблемы в единую историю), но достаточно для отправки.

---

### `complaint_sent`
Срабатывает при каждом нажатии кнопки отправки. **Может быть несколько за сессию** — пользователь может отправить и email, и на портал.

| Свойство | Тип | Описание |
|---|---|---|
| `method` | string | `email` / `portal` |
| `dest` | string | `all_3` / `uk` / `admin` / `ministry` / `governor` / `gis` |
| `problems` | string[] | Список ID выбранных проблем |
| `problems_count` | int | Количество проблем |
| `tone` | string | `formal` / `informal` |
| `length` | string | `short` / `medium` / `long` |

**Для метрики "хотя бы одна отправка":** считать уникальных пользователей с `complaint_sent >= 1`, не сумму событий.
**Для анализа каналов:** разбивка по `dest` — куда чаще всего отправляют.

---

### `share_clicked`
Срабатывает при нажатии кнопки «Поделиться с соседом».

| Свойство | Тип | Описание |
|---|---|---|
| `problems_count` | int | Количество выбранных проблем |

**Использование:** метрика виральности. Показывает долю пользователей, которые хотят привлечь соседей.

---

## Ключевые метрики для дашборда

| Метрика | Формула | Что показывает |
|---|---|---|
| Конверсия в жалобу | `uniq(complaint_sent) / uniq($pageview)` | Основная метрика продукта |
| Вовлечение | `uniq(problem_selected) / uniq($pageview)` | Доля пришедших с намерением |
| ИИ success rate | `uniq(ai_generated) / uniq($pageview)` | Здоровье сервера |
| Пропуск данных | `data_entered[skipped=true] / uniq(data_entered)` | Готовность к персонализации |
| Виральность | `uniq(share_clicked) / uniq(complaint_sent)` | Доля «миссионеров» |

## Bounce rate — почему неинформативен

Классический bounce rate (~65%) в данном продукте не является проблемой. Основной сценарий: пользователь зашёл → выбрал проблему → отправил жалобу → ушёл. Это целевое действие за одну страницу. Вместо bounce rate использовать: **% пользователей без `complaint_sent`**.

## Особенности и нюансы

### Множественные `complaint_sent`
Среднее 2.5 события на пользователя — норма. Каждый способ отправки (email УО, email администрации, портал ГИС ЖКХ и т.д.) пишет отдельное событие. Если пользователь отправил в 3 места — 3 события. Это хорошо.

### `ai_generated` до `step2_reached`
ИИ-генерация запускается автоматически на шаге 1, сразу после выбора первой проблемы. Поэтому `ai_generated` хронологически предшествует `step2_reached`. В PostHog воронку строить без `ai_generated` как обязательного шага.

### Таймаут при зависании ИИ
Если OpenRouter не отвечает, пользователь видит скелетон до ~30 секунд (нет явного таймаута на запрос). После этого появляется шаблонный текст. Кнопки отправки заблокированы на время ожидания.
